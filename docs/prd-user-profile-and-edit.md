# PRD: User Profile & Profile Edit

**Дата**: 2026-05-19
**Статус**: Draft

## Цель

Дать пользователю возможность персонализировать аккаунт — задать имя и аватар, сменить пароль — и видеть эту информацию на странице профиля и в шапке главной страницы.

## Пользовательские сценарии

- Пользователь открывает `/profile` → видит своё имя (или email, если имя не задано), аватар и email
- Пользователь нажимает «Редактировать» → попадает на `/profile/edit`
- Пользователь меняет имя → сохраняет → имя обновляется в профиле и в шапке главной страницы
- Пользователь загружает новый аватар → сохраняет → аватар отображается в профиле и в шапке главной страницы
- Пользователь вводит старый пароль и новый дважды → сохраняет → пароль сменён, прежний токен остаётся валидным
- Пользователь вводит неверный старый пароль → получает ошибку «Неверный текущий пароль»

## В скоупе

- Новые поля в Prisma-модели `User`: `name String?` и `avatarUrl String?`
- API endpoint `GET /users/me` — возвращает `{ id, email, name, avatarUrl }`
- API endpoint `PATCH /users/me` — обновляет `name` и/или `avatarUrl` (JWT-protected)
- API endpoint `POST /users/me/change-password` — принимает `{ oldPassword, newPassword }`, проверяет старый пароль bcrypt-сравнением (JWT-protected)
- Загрузка аватара через уже существующий механизм хранения файлов (`apps/api/uploads/`); аватар сохраняется как обычный файл, в БД хранится путь
- Страница `/profile` (read-only): аватар, имя/email, дата регистрации, кнопка «Редактировать профиль»
- Страница `/profile/edit`: форма с тремя секциями — имя, аватар (drag-and-drop + превью), смена пароля
- Шапка главной страницы (`home-page.tsx`): вместо текста email показывать аватар (если есть) + имя (если есть, иначе email)
- Ссылка на профиль из шапки главной страницы

## Не в скоупе

- Смена email
- Удаление аккаунта
- Публичные страницы профиля других пользователей
- OAuth / сторонние провайдеры
- Обрезка/кадрирование аватара на клиенте

## Технические ограничения

- JWT payload содержит только `sub` (userId) и `email`; после смены имени/аватара токен не обновляется — профиль читается отдельным запросом `GET /users/me`
- Текущий `User`-model не имеет полей `name` и `avatarUrl` — потребуется миграция Prisma
- Загрузка аватара повторяет паттерн Phase 4 (multer, `uploads/` директория); максимальный размер файла — 5 МБ, допустимые типы — `image/jpeg`, `image/png`, `image/webp`
- Смена пароля не инвалидирует существующие JWT-токены (нет blacklist-механизма в текущей архитектуре)

## Критерии готовности

- [ ] `GET /users/me` возвращает `{ id, email, name, avatarUrl }` для авторизованного пользователя
- [ ] `PATCH /users/me` обновляет `name` и `avatarUrl`; невалидные данные возвращают 400
- [ ] `POST /users/me/change-password` возвращает 200 при верном старом пароле и 400 при неверном
- [ ] Загруженный аватар доступен по статическому URL и отображается на странице профиля
- [ ] Страница `/profile` отображает актуальные данные пользователя (имя или email, аватар, дата регистрации)
- [ ] Страница `/profile/edit` позволяет сохранить имя, загрузить аватар и сменить пароль независимо друг от друга
- [ ] После сохранения имени/аватара шапка главной страницы отображает обновлённые данные без перезагрузки
- [ ] Все новые API-эндпоинты покрыты e2e-тестами (успешный путь + ошибки валидации)
- [ ] TypeScript strict-mode проходит без ошибок (`npm run typecheck`)
