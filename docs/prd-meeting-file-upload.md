# PRD: Meeting File Upload

**Дата**: 2026-05-14
**Статус**: Draft

## Цель

Позволить пользователям прикреплять файлы (записи встречи, транскрипты, документы) к конкретной встрече — для последующего хранения и обработки. Файлы отображаются внутри страницы встречи и доступны всем участникам.

## Пользовательские сценарии

- Пользователь открывает встречу и нажимает «Загрузить файл» → файл сохраняется и появляется в списке вложений встречи
- Пользователь видит список файлов встречи → может скачать любой файл по клику
- Владелец встречи удаляет прикреплённый файл → файл исчезает из списка и удаляется из хранилища
- Пользователь загружает файл недопустимого типа или размера → получает понятное сообщение об ошибке

## В скоупе

- Загрузка файлов через `multipart/form-data` на POST `/meetings/:id/files`
- Хранение файлов локально на диске (путь: `uploads/meetings/{meetingId}/{fileId}/{filename}`)
- Сохранение метаданных файла в БД (id, meetingId, originalName, mimeType, size, filePath, uploadedAt)
- Получение списка файлов встречи через GET `/meetings/:id/files`
- Скачивание файла через GET `/meetings/:id/files/:fileId/download` (стримится с диска через NestJS `StreamableFile`)
- Удаление файла владельцем встречи через DELETE `/meetings/:id/files/:fileId`
- UI-панель «Файлы» внутри страницы встречи: список файлов с именем, размером, датой загрузки
- Кнопка загрузки с drag-and-drop зоной и прогресс-баром
- Иконка типа файла по MIME-type

## Не в скоупе

- Автоматическая обработка / транскрибирование файлов (следующая итерация)
- Превью файлов внутри интерфейса (PDF viewer, видеоплеер)
- Версионирование файлов
- Права доступа на уровне отдельного файла (только уровень встречи)
- Поддержка загрузки нескольких файлов одновременно одним запросом

## Технические ограничения

- **Monorepo**: NestJS backend (`apps/api`), Next.js 15 frontend (`apps/web`); CQRS через CommandBus/QueryBus
- **Prisma + PostgreSQL**: схема расширяется добавлением модели `MeetingFile`; миграция обязательна
- **Максимальный размер файла**: 100 МБ (ограничение Fastify multipart)
- **Допустимые типы**: видео (mp4, mov, webm), аудио (mp3, m4a, wav), документы (pdf, docx, xlsx, pptx, txt)
- **Авторизация**: только владелец встречи может удалять файлы; просмотр и скачивание — для всех участников (проверка через `JwtGuard` + ownership check)
- **Локальное хранилище**: директория `uploads/` в корне `apps/api`; путь конфигурируется через переменную окружения `UPLOAD_DIR` (по умолчанию `./uploads`)
- **Директория создаётся автоматически** при старте приложения, если не существует

## Критерии готовности

- [ ] Модель `MeetingFile` добавлена в Prisma-схему и применена миграция
- [ ] POST `/meetings/:id/files` принимает файл, сохраняет на диск и записывает метаданные в БД
- [ ] GET `/meetings/:id/files` возвращает список файлов встречи с метаданными
- [ ] GET `/meetings/:id/files/:fileId/download` стримит файл с диска с корректным `Content-Disposition`
- [ ] DELETE `/meetings/:id/files/:fileId` удаляет файл с диска и из БД; доступно только владельцу встречи
- [ ] Загрузка файла > 100 МБ или недопустимого типа возвращает 400 с описанием ошибки
- [ ] Страница встречи содержит вкладку/секцию «Файлы» с drag-and-drop зоной и списком файлов
- [ ] Прогресс-бар отображается во время загрузки файла
- [ ] Клик по файлу запускает скачивание через эндпоинт `/download`
- [ ] Кнопка удаления файла видна только владельцу встречи и работает без перезагрузки страницы
- [ ] E2E тест покрывает сценарий: загрузка → отображение → скачивание → удаление
